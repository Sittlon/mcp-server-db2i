# Configuration reference

`mcp-server-db2i` reads a single JSON file. Loader search order:

1. `--config <path>` (CLI flag)
2. `./config/config.json`
3. `./config.json`
4. `~/.config/mcp-server-db2i/config.json`

The schema is enforced at startup with [zod](https://zod.dev). Validation errors are reported with field paths so you can fix them quickly. A JSON-schema file (`config.schema.json`) is generated by `npm run gen:schema` and referenced via `"$schema": "./config.schema.json"` for editor IntelliSense.

Environment variables of the form `${VARNAME}` are interpolated at load time anywhere in the file (string values only).

## Top-level shape

```jsonc
{
  "$schema": "./config.schema.json",
  "connection": { /* see § Connection */ },
  "security":   { /* see § Security */ },
  "schema":     { /* see § Schema annotations */ },
  "templates":  { /* see § Templates */ }
}
```

## Connection

```jsonc
{
  "connection": {
    "driver": "odbc",                 // "odbc" | "jt400"
    "odbc":  { "connectionString": "DSN=AS400;UID=READER;PWD=${DB2I_PASSWORD}" },
    "jt400": { "host": "host.example.com", "user": "READER", "password": "${DB2I_PASSWORD}", "naming": "system" }
  }
}
```

- Exactly one of `odbc` / `jt400` is required (matching `driver`).
- For ODBC, install IBM i Access Client Solutions or unixODBC + the IBM i ODBC driver.
- For jt400, install the optional `jt400` npm package (`npm install jt400`) and a JRE.
- `naming`: `"system"` uses `LIBRARY/FILE`, `"sql"` uses `SCHEMA.TABLE`. Default is `system`.

## Security

```jsonc
{
  "security": {
    "defaultRole": "reader",
    "roles": {
      "reader": {
        "allowRawSelect": false,
        "allowedTemplates": ["*"],
        "maxRows": 500,
        "allowedSchemas": ["SAMPLE"]
      },
      "analyst": {
        "allowRawSelect": true,
        "allowedTemplates": ["*"],
        "maxRows": 5000,
        "allowedSchemas": ["SAMPLE", "QGPL"],
        "blockedTables": ["USERS", "CREDENTIALS"]
      }
    }
  }
}
```

| Field                     | Type      | Notes                                                          |
|---------------------------|-----------|----------------------------------------------------------------|
| `defaultRole`             | string    | Must exist as a key in `roles`.                                |
| `roles[name].allowRawSelect` | boolean | When `false`, the `db2i_query` tool is not advertised.        |
| `roles[name].allowedTemplates` | string[] | Reserved; currently `["*"]` allows any template the role is listed in. |
| `roles[name].maxRows`     | number    | Positive integer; appended as `FETCH FIRST N ROWS ONLY`.       |
| `roles[name].allowedSchemas` | string[] | `["*"]` disables schema gating.                              |
| `roles[name].blockedTables` | string[] | Optional; matched against bare (unqualified) table names.    |

Override the active role at runtime with `--role <name>`.

## Schema annotations

```jsonc
{
  "schema": {
    "tables": {
      "SAMPLE.ORDERS": {
        "description": "Sales orders — one row per line item",
        "columns": {
          "ORDNR": { "description": "Order number",     "type": "DECIMAL(10,0)" },
          "PRICE": { "description": "Unit price (EUR)", "type": "DECIMAL(9,2)" }
        }
      }
    }
  }
}
```

These annotations are injected into the LLM-facing tool descriptions so the model sees `PRICE — Unit price (EUR)` instead of guessing what `PRICE` means. Use `npm run introspect -- --schema MYLIB` to bootstrap, then fill in descriptions (manually or via `tools/annotate/`).

## Templates

```jsonc
{
  "templates": {
    "orders_by_region": {
      "description": "Order volume and revenue grouped by region for a date range",
      "parameters": {
        "from_date": { "type": "date",    "description": "Start date (YYYY-MM-DD)" },
        "to_date":   { "type": "date",    "description": "End date (YYYY-MM-DD)" },
        "limit":     { "type": "integer", "description": "Max rows", "default": 100 }
      },
      "sql": "SELECT REGION, SUM(QTY) AS units FROM SAMPLE.ORDERS WHERE ODATE BETWEEN ? AND ? GROUP BY REGION FETCH FIRST ? ROWS ONLY",
      "allowedRoles": ["reader", "analyst"]
    }
  }
}
```

- Parameters are passed positionally in the order they appear in `parameters`. Each `?` in `sql` consumes one parameter.
- Supported `type` values: `string`, `integer`, `date`, `decimal`. (Type coercion is the connector's job; this metadata is currently informational for the LLM.)
- `default` values are used when the LLM omits a parameter.
- `allowedRoles` is validated at config load time — every role must exist in `security.roles`.
